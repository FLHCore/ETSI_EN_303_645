## 6.1 – 消費者資料保護條款 (DATA PROTECTION PROVISIONS FOR CONSUMER)

### 條款 6.3 (Provision 6.3)

已同意處理其個人數據的消費者，必須具備隨時撤回該同意的能力。

#### 最低要求 (Minimum Requirements)

* 設備、其配套應用程式及其他相關服務（如適用）必須允許使用者**隨時撤回**對其個人數據收集與處理的同意，即便他們先前已經給予同意。

#### 佐證證據 (Supporting Evidence)

* 開發商應提供證據（例如：GUI 中撤回功能的截圖、透過產品網頁提交撤回請求、解釋如何撤回使用者資訊處理的指引文件等），顯示使用者可以隨時撤回對其個人數據收集與處理的同意。

#### 評估 (Assessment)

* 評估人員將檢查證據，以判斷使用者是否能夠隨時撤回對其個人數據收集與處理的同意，即便他們先前已經給予同意。

---

### 開發者深度導讀 (針對 Gateway / App / Cloud 開發商)

#### 本章節核心要求與意涵

本條款即是隱私法規中常見的**「撤回同意權 (Right to Withdraw Consent)」**，在實作上常被稱為 **Opt-out (退出)** 機制。

1.  **雙向開關 (Toggle)**：
    * 同意不能是單行道。如果您在安裝時問了使用者「是否同意參加改善計畫」，事後必須在「設定」選單中提供一個開關讓使用者能反悔。
    * **易用性原則**：撤回同意的難度不應高於給予同意的難度。如果當初只是「點一下同意」，那撤回時也不能要求使用者「寫信給客服」或「填寫複雜表單」。
2.  **生效的即時性**：
    * 雖然規範未定義「立即」，但技術上應確保當使用者將開關切換為 OFF 時，設備端應**立即停止**上傳相關數據。

#### 具體落地實現方案建議

* **系統設計與狀態同步**：
    * **本地與雲端同步**：
        * 當使用者在 App 端關閉權限時，App 需同時做兩件事：
            1.  停止本地 SDK 的資料收集。
            2.  呼叫 API 通知雲端更新該用戶的 `consent_status`。
    * **Granular Control (細粒度控制)**：
        * 建議在「隱私設定」頁面提供細項開關，而非只有一個總開關。
        * 例如：
            * [ ] 分享使用數據 (Analytics)
            * [ ] 分享位置資訊 (Location)
            * [ ] 接收行銷資訊 (Marketing)

* **證據收集策略**：
    * **「設定」頁面截圖**：
        * 這是最核心的證據。截取 App 或 Web GUI 中的「隱私設定 (Privacy Settings)」頁面，展示那些可以被關閉的 Toggle Button。
    * **操作指引**：
        * 截取 FAQ 或說明書中「如何關閉資料收集？」的章節。

* **測試與驗證**：
    * **流量截斷測試**：
        * **步驟 1**：開啟「分享使用數據」。確認 Wireshark 或 Proxy 軟體 (如 Charles) 能抓到上傳 Log 的封包。
        * **步驟 2**：關閉該選項。
        * **步驟 3**：繼續操作 App。
        * **預期結果**：不應再看到任何 Log 上傳的封包。如果關閉後還在傳，就是嚴重的 Bug。
