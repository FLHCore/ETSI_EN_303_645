## 5.3 – 保持軟體更新 (KEEP SOFTWARE UPDATED)

### 條款 5.3-3 (Provision 5.3-3)

更新操作必須對使用者而言是簡單的 (simple)。

#### 最低要求 (Minimum Requirements)

* 必須提供說明 (instructions) 給使用者，解釋如何使用設備的更新機制來為其設備應用更新 。

#### 佐證證據 (Supporting Evidence)

* 開發商應提供證據（例如：使用者手冊、影片等），顯示已提供說明給使用者，解釋如何為設備應用更新 。
* 可接受的範例包括但不限於 ：
    * 如何切換自動更新開關 。
    * 如何在 GUI 和/或行動應用程式內應用更新 。
    * 如何從開發商網站下載韌體並將其安裝在設備上 。

#### 評估 (Assessment)

* 評估人員將檢查是否已提供關於如何為設備應用更新的說明給使用者 。

---

### 開發者深度導讀 (針對 Gateway / App / Cloud 開發商)

#### 本章節核心要求與意涵

本條款關注的是 **UX (使用者體驗)** 與 **文件完整性**。

1.  **「簡單」的定義**：雖然「簡單」是主觀形容詞，但在合規審查中，它轉化為「是否有足夠的指引」。如果更新流程需要使用者具備工程師等級的技能（例如：透過 Serial Port 刷機、需要輸入複雜指令），則不符合此要求。
2.  **文件即功能**：即便您的 UI 設計得再直覺，若缺乏對應的說明文件（User Manual 或 In-App Help），此條款仍會判定失敗。重點在於**告知**使用者如何操作。

#### 具體落地實現方案建議

* **系統設計與 UI 調整**：
    * **一鍵更新 (One-Click Update)**：這是最符合「簡單」定義的設計。當偵測到新版本時，App 或 Web GUI 彈出視窗，使用者只需點擊「立即更新」，其餘下載、驗證、安裝、重開機流程皆由系統自動完成。
    * **狀態回饋**：更新過程中應顯示進度條或明確的狀態（如「更新中，請勿斷電」），避免使用者因以為當機而拔除電源。

* **證據收集策略**：
    * **使用者手冊截圖**：這是最標準的證據。請翻閱您的產品說明書 (User Manual)，找到「韌體更新 (Firmware Update)」的章節，截取該頁面。
    * **線上說明中心 (Help Center)**：如果您的產品主要依賴 App，請截取 App 內的「常見問題 (FAQ)」或「幫助」頁面中，關於如何更新的說明畫面。
    * **影片示範**：雖然截圖通常足夠，但若更新流程較特殊，錄製一段 30 秒的操作影片也是很好的佐證。

* **測試與驗證**：
    * **文件審查 (Doc Review)**：QA 或技術寫作人員 (Technical Writer) 需確認手冊中的步驟與實際 App/Web 介面是否一致（例如：按鈕名稱是否改變、選單路徑是否正確）。
    * **可用性測試**：找一位非技術背景的同事，給他說明書，看他能否順利完成更新操作。
