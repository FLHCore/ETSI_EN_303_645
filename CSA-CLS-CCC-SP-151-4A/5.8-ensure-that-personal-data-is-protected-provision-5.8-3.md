## 5.8 – 確保個人數據受保護 (ENSURE THAT PERSONAL DATA IS PROTECTED)

### 條款 5.8-3 (Provision 5.8-3)

設備的所有外部感測能力 (external sensing capabilities) 必須以對使用者清晰且透明的易存取方式進行記錄與說明。

#### 最低要求 (Minimum Requirements)

* 設備的所有外部感測能力必須有完善的文件記錄，且使用者應能輕易存取這些資訊（例如：透過提示、文件、警示等）。
* 這也包含那些可能**預設停用**的外部感測能力（例如：麥克風、藍牙等）。
* (設備的外部感測能力包括使用光學、聲學、生物特徵或定位感測器)。

#### 佐證證據 (Supporting Evidence)

* 開發商應宣告設備是否具備任何外部感測能力，包括那些可能預設停用的感測能力。
* 開發商應提供證據（例如：截圖、附件、使用者指引文件等），顯示設備的外部感測能力是如何傳達給使用者的。
* 具備外部感測能力的設備範例包括但不限於：
    * 具備音訊麥克風等外部感測能力的 IP 攝影機設備。
    * 具備 GPS 等外部感測能力的智慧手錶設備。

#### 評估 (Assessment)

* 評估人員將檢查開發商是否宣告設備具備任何外部感測能力。
* 若有，評估人員將檢查關於設備外部感測能力的資訊是否已提供給使用者。

### 開發者深度導讀 (針對 Gateway / App / Cloud 開發商)

#### 本章節核心要求與意涵

本條款的核心在於**「硬體能力的誠實揭露」**。這是一個針對隱私權的信任測試。

1.  **隱藏硬體的風險**：
    * 過去曾有智慧音箱或家庭中樞 (Home Hub) 被發現內建麥克風，但說明書上沒寫，導致輿論譁然。即使軟體尚未啟用該功能，消費者仍會擔心該硬體可能被惡意軟體啟用進行竊聽。
    * **合規要求**：只要 PCB 板上有焊上感測器（如 MEMS Mic, Light Sensor, GPS），無論軟體是否使用，都必須在使用者可見的文件或介面上明確告知。
2.  **範圍定義**：
    * **光學 (Optical)**：攝影機、光感應器、LiDAR。
    * **聲學 (Acoustic)**：麥克風。
    * **生物特徵 (Biometric)**：指紋辨識、人臉辨識模組。
    * **定位 (Location)**：GPS、GLONASS，甚至是用於定位的 Wi-Fi/Bluetooth 掃描功能。

#### 具體落地實現方案建議

* **系統設計與使用者溝通**：
    * **隱私規格表 (Privacy Spec Sheet)**：
        * 在產品包裝或官網規格頁，增加一個專屬欄位：「感測器列表」。
        * 範例：`Microphone: Yes (Disabled by default)`。
    * **App 初始化流程 (Onboarding)**：
        * 當使用者首次設定設備時，App 應彈出說明：「本設備內建麥克風，用於未來的語音功能，目前預設為關閉。若要啟用，請至設定頁面開啟。」
    * **硬體隱私開關 (Hardware Kill Switch)**：
        * 對於高隱私風險的攝影機或麥克風，最佳實務是設計實體滑蓋或斷路開關，讓使用者能物理遮蔽。

* **證據收集策略**：
    * **BOM 表與說明書對照**：
        * PM 需檢查硬體 BOM 表，圈出所有感測器元件。
        * 對照 User Manual 或官網規格頁截圖，證明這些元件都有被列出來。
    * **軟體狀態截圖**：
        * 若感測器預設是關閉的，截取 App 設定頁面，顯示該功能的開關預設是 "OFF"。

* **測試與驗證**：
    * **拆機檢查 (Teardown)**：
        * 雖然評估人員主要看文件，但內部 QA 或稽核應實際拆開機殼，檢查 PCB 上是否有未被記錄的感測器晶片（例如溫濕度計雖然敏感度較低，但嚴格來說也算環境感測）。
    * **功能啟用測試**：
        * 測試當使用者嘗試啟用該感測器（如開啟語音助理）時，系統是否會再次跳出隱私同意書 (Privacy Consent)。
