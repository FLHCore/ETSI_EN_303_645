## 5.1 – 禁用通用預設密碼 (NO UNIVERSAL DEFAULT PASSWORDS)

### 條款 5.1-4 (Provision 5.1-4)

當使用者能夠對設備進行身分驗證時，設備必須提供使用者或管理者一個簡單的機制來更改所使用的認證值（Authentication Value，如密碼）。

* **a.** 家庭閘道器 (HG) 必須允許管理者設定 Wi-Fi® 密碼。
* **b.** HG 必須提供本地管理者一個簡單的機制來更改 Wi-Fi® 密碼。
* **c.** HG 必須提供管理者一個簡單的機制來更改管理者密碼（本地端對本地端，遠端對遠端）。

#### 最低要求 (Minimum Requirements)

* 對於具有圖形使用者介面 (GUI) 的設備，設定和更改 Wi-Fi/管理者密碼的過程必須**容易執行**。
* 對於沒有 GUI 的設備，為了迎合沒有技術背景的管理者，必須透過使用者手冊、教學影片等方式，提供詳細的指引給使用者，引導他們完成設定和更改 Wi-Fi/管理者密碼的過程。

#### 佐證證據 (Supporting Evidence)

* 開發商應提供證據（例如：截圖、使用者手冊、影片等），詳述管理者如何在所有可設定和更改密碼的設備平台（例如：設備管理入口網站、行動應用程式、本地端對本地端、遠端對遠端等）上設定和更改 Wi-Fi/管理者密碼（例如：重設密碼、更改 PIN 碼等）。
* 開發商亦應提供**截圖**，顯示當管理者試圖設定一個**不符合**條款 5.1-1 最低複雜度要求（例如：密碼長度太短、密碼未達複雜度要求等）的密碼時，所出現的**失敗訊息 (Failure Message)**。

#### 評估 (Assessment)

* 評估人員將檢查設備的 GUI 是否允許使用者設定和更改密碼。
* 對於沒有 GUI 的設備，評估人員將檢查是否已提供說明給管理者，解釋如何設定和更改設備密碼。
* 若認證值為密碼，評估人員將檢查所提供的證據，確保該介面會**拒絕**不符合最低密碼要求的更改嘗試。

### 開發者深度導讀 (針對 Gateway / App / Cloud 開發商)

#### 本章節核心要求與意涵

本條款看似簡單（即「要有改密碼功能」），但其實際審查重點在於**易用性 (Usability)** 與 **強制合規性 (Enforcement)** 的結合。

1.  **使用者體驗 (UX) 要求**：「簡單 (Simple)」與「容易執行 (Easy to execute)」是主觀但關鍵的要求。這意味著「更改密碼」的功能不能埋藏在多層選單深處（例如：進階設定 -> 系統工具 -> 管理 -> 密碼）。它應該在明顯的位置，或者有設定精靈 (Wizard) 引導。
2.  **範圍覆蓋**：此條款同時涵蓋 **Admin 登入密碼** 與 **Wi-Fi SSID 密碼**。兩者都必須具備易於修改的介面。
3.  **與 5.1-1 的連動驗證**：這是最關鍵的技術點。開發者不能只做一個「輸入框」，必須將 5.1-1 定義的複雜度規則（10 碼以上、混合字元等）實作在前端與後端的驗證邏輯中。

#### 具體落地實現方案建議

* **系統設計與 UI 調整**：
    * **即時回饋機制**：在密碼輸入框下方加入動態檢查列表（Checklist）。
        * [ ] 長度大於 10 碼
        * [ ] 包含大寫英文
        * [ ] 包含數字
        * 當使用者輸入時，動態將符合的項目打勾。這不僅提升 UX，也能直接證明符合「簡單機制」與「複雜度要求」。
    * **錯誤訊息明確化**：當驗證失敗時，錯誤訊息不能只寫「密碼錯誤」或「格式不符」。必須明確提示：「密碼長度不足，請輸入至少 10 個字元」或「請包含至少一個數字」。這正是佐證證據中要求截圖的重點。

* **證據收集策略**：
    * **「失敗畫面」截圖是必備項**：許多開發者會忘記這一點，只截取了「密碼修改成功」的畫面。請務必刻意輸入一組弱密碼（如 "123"），觸發錯誤視窗，並截取該畫面作為證據。
    * **多平台覆蓋**：如果您的產品同時有 Web GUI 和 Mobile App，兩邊的「修改密碼」介面都必須截圖提供。不能只提供其中一種。

* **測試與驗證**：
    * **邊界測試**：QA 應測試剛好 9 碼（應失敗）與剛好 10 碼（應成功）的情況。
    * **一致性測試**：確認 Web 端改了密碼後，App 端是否會被登出或要求重新登入（Session 管理），雖然此條款未明說，但這屬於「更改認證值」後的預期安全行為。
